## 维度一：API 网关与流量治理 (Gateway + Sentinel)

### Q1：你们的微服务网关鉴权是怎么设计的？如何保证所有服务的入口安全？

**Situation（情境）：** 系统拆分为 8 个业务微服务，前端包括微信小程序、商家端、管理后台三个入口。如果每个服务各自做鉴权，会出现鉴权逻辑分散、白名单管理混乱、token 格式不统一等问题。一旦某个服务鉴权出现漏洞，整个系统都有被越权访问的风险。

**Task（任务）：** 设计一套统一的网关层鉴权方案，在请求进入微服务集群之前完成身份校验，同时兼顾公开接口（登录、支付回调、商家入驻等）的放行需求。

**Action（行动）：**
1. **统一鉴权收敛到 Gateway 层**。开发 `AuthGlobalFilter`（实现 `GlobalFilter + Ordered` 接口），所有请求必须经过此过滤器。
2. **多来源 Token 提取**。优先级：`Authorization` 请求头（Bearer 格式） > URL query 参数 `token`。兼容小程序和 H5 不同的传参习惯。
3. **白名单分级管理**。将公开路径分为固定白名单（登录、注册、支付回调、Swagger 文档）和动态可配置白名单（后续计划接入 Nacos Config 实现热更新）。特别注意大小写匹配问题（如 `/wxPay` 和 `/pay` 都需覆盖）。
4. **鉴权通过后透传身份**。将验证通过的 token 放入自定义请求头 `X-Auth-Token`，下游服务无需再解析 token，直接通过该 header 识别用户身份。
5. **OPTIONS 预检请求直接放行**，避免 CORS 预检被误拦截。
6. 失败统一返回 401 + JSON 格式错误信息，前端可统一拦截处理。

```java
// Gateway 将 token 透传给下游服务
ServerHttpRequest mutatedRequest = request.mutate()
    .header("X-Auth-Token", token)
    .build();
```

**Result（结果）：** 8 个微服务全部剥离了入口鉴权逻辑，只需关注自身业务。白名单集中在 Gateway 一个文件管理，新增公开接口只需改一处。下游服务通过 `X-Auth-Token` 统一获取用户身份，实现了认证与业务的彻底解耦。

### Q2：大促期间突发流量洪峰，你们的网关怎么扛？

**Situation（情境）：** 在外卖平台午高峰（11:00-13:00）或促销活动上线瞬间，订单服务可能面临瞬时流量放大 10 倍以上的压力。如果不做限流，后端服务会被打满导致雪崩。

**Task（任务）：** 在网关层建立统一的流量治理机制，保护后端微服务不被突发流量击垮。

**Action（行动）：**
1. **Gateway 选型层面**：采用 Spring Cloud Gateway（基于 Spring WebFlux 的响应式异步模型），相比传统 Zuul 1.x 的阻塞 IO 模型，同等硬件资源下可承载更高的并发连接数。
2. **路由层面配置 StripPrefix**，确保转发路径精确匹配下游 Controller，避免无效请求浪费后端资源。
3. **全局 CORS 统一处理**，在 Gateway 层配置 `globalcors`，下游服务无需再单独处理跨域，减少了一层网络开销。
4. **限流策略（架构设计层面）**：基于 Redis + Gateway 的 `RequestRateLimiter` 实现令牌桶限流，以 IP + 用户维度做双维度限流，单 IP 每秒限流 100 次，单用户每秒限流 50 次。
5. **下游服务自我保护**：各微服务独立配置 Sentinel 规则，对下单、支付等核心接口配置 QPS 阈值 + 热点参数限流。当流量超过阈值时返回降级提示而非直接报错。
6. **健康检查端点收敛**：仅暴露 `/health` 和 `/prometheus`，关闭 `/env`、`/heapdump` 等敏感端点，防止外部利用健康检查接口消耗资源。

**Result（结果）：** 网关层作为第一道防线拦截了大部分无效和超限请求，后端服务专注于处理合法流量。通过 Gateway + Sentinel 的双层防护体系，确保核心服务在流量洪峰期间依然稳定可用。

## 维度二：分布式一致性与事务流转 (Seata + RocketMQ)

### Q3：用户下单场景涉及订单、库存、优惠券三个服务，如何保证数据一致性？

**Situation（情境）：** 外卖系统的核心下单链路横跨多个微服务：订单服务创建订单记录 -> 商品服务扣减库存 -> 促销服务核销优惠券。三个操作分布在三个独立的数据库上，任何一步失败都会导致数据不一致（比如订单创建了但库存没扣，用户没付款商品就少了一份）。

**Task（任务）：** 保证跨服务、跨数据库的下单操作要么全部成功，要么全部回滚，不能有中间态。

**Action（行动）：**
1. **强一致性场景采用 Seata AT 模式**。在订单创建的入口方法标注 `@GlobalTransactional`，由 Seata TC 协调全局事务。
   - 一阶段：各 RM（Resource Manager）执行业务 SQL，同时自动生成 undo log（前置镜像 + 后置镜像），提交本地事务
   - 二阶段提交：异步删除 undo log
   - 二阶段回滚：利用 undo log 的前置镜像恢复数据
   - 业务代码零侵入，无需手动写补偿逻辑

2. **最终一致性场景采用 RocketMQ 事务消息**。对于非核心链路（如发送通知、写操作日志、更新统计数据），使用 RocketMQ 事务消息保证最终送达：
   - 发送 Half 消息（对消费者不可见）
   - 执行本地事务（如更新订单状态）
   - 根据本地事务结果向 Broker 发送 Commit / Rollback
   - Broker 回查机制兜底：长时间未确认时回调 `checkLocalTransaction` 检查状态

3. **一致性级别权衡策略**：
   - **资金相关（余额扣减、支付退款）** → 强一致性（Seata），宁可慢不能错
   - **核心业务（库存扣减、订单创建）** → 强一致性（Seata）
   - **非核心链路（通知、日志、统计）** → 最终一致性（RocketMQ），允许短暂延迟

**Result（结果）：** 通过 Seata 保障了资金和核心业务的强一致性，通过 RocketMQ 保障了非核心链路的最终一致性。两种方案各取所长，既保证了数据正确，又避免了所有操作都走分布式事务导致的性能瓶颈。

### Q4：如果 Seata 的 TC（事务协调器）宕机了，正在进行中的全局事务怎么办？

**Situation（情境）：** Seata 架构中 TC 是全局事务的协调者，负责向各 RM 发送 Commit 或 Rollback 指令。如果 TC 在执行全局事务的中间阶段宕机，部分 RM 已执行完本地事务但没收到二阶段指令，数据会处于不一致的中间态。

**Task（任务）：** 设计兜底方案，确保 TC 异常情况下系统能自动恢复或人工介入修复。

**Action（行动）：**
1. **TC 高可用部署**：Seata Server 以集群模式部署，注册到 Nacos，通过负载均衡避免单点故障。
2. **undo log 持久化兜底**：一阶段执行后 undo log 已持久化到业务数据库中。即使 TC 宕机，重启后可通过 undo log 手动回滚。
3. **全局事务状态表**：`global_table`、`branch_table`、`lock_table` 三张表记录了所有进行中的事务状态。重启 TC 后，根据这些表的状态继续执行未完成的二阶段操作。
4. **超时自动回滚机制**：配置全局事务超时时间（默认 60 秒），超过时限的分支事务自动触发回滚。
5. **人工介入工具**：提供管理后台查询异常事务的界面，展示 `global_table` 中的 stuck 事务，支持手动触发提交或回滚。
6. **监控告警**：通过 Prometheus 监控 Seata Server 健康状态，异常时及时告警。

**Result（结果）：** 通过 TC 集群 + undo log 持久化 + 超时自动回滚的三层保护，确保即使 TC 宕机，系统也能在可接受的时间窗口内自动恢复，不会出现永久不一致的数据。

## 维度三：高并发支付与资金安全 (Redis 核心场景)

### Q5：微信支付回调如何保证资金安全？怎么防止伪造回调和重复回调？

**Situation（情境）：** 微信支付成功后，微信服务器会异步回调我们后端的 `wxNotify` 接口。这里存在两个致命安全问题：第一，恶意攻击者可以伪造微信支付回调请求，如果验签不严格，可能导致未付款的订单被标记为已支付；第二，微信在网络抖动时会重复发送同一笔订单的回调，如果不做幂等控制，用户可能被重复加钱、重复发货。

**Task（任务）：** 构建支付回调的双重安全防线——验签 + 幂等控制。

**Action（行动）：**

**第一道防线：签名验证**
1. 微信回调内容为 XML 格式，解析后提取所有参数到 Map
2. 使用微信支付 V2 的 MD5 签名算法，将参数按字典序排序拼接，末尾拼接商户密钥：`text + "&key=" + mchKey`
3. 将计算出的签名与回调中的 `sign` 字段比对，不一致则拒绝并返回 FAIL
4. **关键细节**：签名原始串的拼接格式必须与微信官方一致，多一个少一个字符都会导致验签失败。特别注意 `&key=` 前缀不能遗漏。

**第二道防线：Redis 分布式锁幂等控制**
```java
String lockKey = "pay:callback:lock:" + outTradeNo;
Boolean isLocked = stringRedisTemplate.opsForValue()
    .setIfAbsent(lockKey, "1", 3, TimeUnit.MINUTES);
if (Boolean.FALSE.equals(isLocked)) {
    // 同一订单正在处理或已处理，幂等拦截
    resXml = "<xml><return_code><![CDATA[SUCCESS]]></return_code>...</xml>";
    return;
}
try {
    // 核心业务：检查订单状态 → 更新订单为已支付
    //          → 增加商家余额 → 发放优惠券 → 发送通知
} finally {
    // 锁保留至自然过期，不主动删除
    // 防止微信在高并发场景下重复回调
}
```

**锁策略设计考量：**
- `setIfAbsent`（SETNX）是原子操作，不存在并发竞争
- 过期时间 3 分钟，足以覆盖正常业务处理时长，同时防止死锁
- `finally` 中不主动删锁：因为微信可能在高并发下重复发送同一笔回调，删除后第二个请求会再次进入业务逻辑
- 幂等拦截后直接回复 SUCCESS，微信收到成功就不会继续重试

**Result（结果）：** 验签防线确保了只有微信官方签发的真实回调才能通过，分布式锁防线确保了同一笔订单在任何时间点只被处理一次。两道防线缺一不可，共同构成了支付资金安全的基石。

### Q6：除了支付回调，Redis 在你们项目中还承载了哪些核心场景？

**Situation（情境）：** 一个成熟的微服务系统，Redis 不能只做简单的 KV 缓存，需要承担更多核心职责。

**Task（任务）：** 梳理项目中 Redis 的关键应用场景，确保每个场景的选型和实现都经得起生产环境的考验。

**Action（行动）：**

| 场景 | 数据结构 | 设计要点 |
|------|---------|---------|
| **用户 Session** | String（Key: `LOGIN_USER:{token}`） | 登录成功写入，设置 7 天 TTL；每次请求续期，活跃用户不会被踢出 |
| **支付幂等锁** | String（Key: `pay:callback:lock:{outTradeNo}`） | SETNX 实现，3 分钟过期，finally 中不主动删锁 |
| **短信验证码** | String（Key: `sms:code:{mobile}`） | 5 分钟过期，发送时覆盖旧值，防暴力爆破 |
| **购物车缓存** | Hash（Key: `cart:{userId}`, Field: `goodsId`, Value: 数量） | 用户加购时先写 Redis，下单时同步到 DB |
| **登录重试限制** | String（Key: `login:lock:{ip}`） | 连续输错密码 5 次后锁定 15 分钟，防暴力破解 |
| **分布式 ID 生成辅助** | String（`INCR` 原子递增） | 辅助生成业务编号（如订单号、排队号） |

**设计原则：**
1. **所有 Key 统一前缀**，便于运维排查和管理
2. **过期时间必须有**，防止内存泄漏
3. **大 Key 预警**：购物车超过 100 个商品时异步刷盘
4. **容灾降级**：Redis 不可用时，Session 查询降级为查 DB

**Result（结果）：** Redis 在项目中承担了认证、幂等、缓存、限流等多重职责。每个场景根据业务特性选择了合适的数据结构，配合 TTL、前缀规范、容灾降级策略，确保在生产环境下稳定运行。

## 维度四：服务治理与路由发现 (Nacos)

### Q7：为什么选择 Nacos 而不是 Eureka？在生产环境中你们是怎么用的？

**Situation（情境）：** 微服务架构中注册中心是核心基础设施，选型直接影响到整个集群的可用性和运维成本。早期很多团队用 Eureka，但随着业务规模扩大和技术栈演进，需要做出更合理的选择。

**Task（任务）：** 选择并落地一个能满足高可用、动态配置、运维友好的服务注册中心。

**Action（行动）：**

**选型考量：**

| 维度 | Eureka | Nacos |
|------|--------|-------|
| 维护状态 | 2.x 停止更新，Spring Cloud 官方已放弃 | 阿里巴巴持续维护，社区活跃 |
| 功能范围 | 仅服务注册 | 注册中心 + 配置中心二合一 |
| 一致性模型 | 仅支持 AP | 支持 AP（默认）和 CP（Raft）切换 |
| 配置管理 | 需搭配 Spring Cloud Config | 原生支持，支持动态刷新 |
| 中文社区 | 弱 | 强，文档完善 |

**生产落地方案：**
1. **单机开发 + 集群生产**：本地开发用 `startup.cmd -m standalone` 单机模式，生产环境部署 3 节点 Nacos 集群，通过 Raft 协议保证数据一致性。
2. **服务分组管理**：不同环境通过 `namespace` 隔离（dev、test、prod），同环境不同业务线通过 `group` 分组。
3. **健康检查机制**：Nacos 通过心跳检测（默认 5 秒间隔）判断服务健康状态。15 秒无心跳标记不健康，30 秒剔除。
4. **配置热更新**：接入 Nacos Config，配置文件变更无需重启服务，通过 `@RefreshScope` 注解实现动态刷新。

**Result（结果）：** 一套 Nacos 同时解决了服务注册和配置管理两大痛点，减少了运维组件数量。AP 模式下服务发现的延迟在毫秒级，满足了大部分业务场景的实时性要求。

### Q8：服务上下线时，如何确保流量不会打到已经挂掉的节点？

**Situation（情境）：** 服务重启、发布、扩容时，如果负载均衡器仍将请求路由到已下线的实例，会导致大量请求失败，直接影响用户体验和交易成功率。

**Task（任务）：** 设计一套服务上下线时的流量切换机制，确保零宕机发布和优雅上下线。

**Action（行动）：**

**服务下线（优雅停机）：**
1. **主动注销**：服务关闭前（通过 Spring Boot Actuator `/shutdown` 端点），主动向 Nacos 发起注销请求，标记自己为不健康状态
2. **被动检测**：Nacos 心跳检测发现服务超过 15 秒无心跳，自动将其标记为不健康，从可用实例列表中移除
3. **LoadBalancer 缓存刷新**：客户端（Feign + LoadBalancer）缓存的服务实例列表有 TTL（默认 30 秒），超时后重新从 Nacos 拉取，自动剔除不健康实例
4. **宽限期设计**：服务接收到关闭信号后，先注销注册 → 停止接收新请求 → 等待已有请求处理完成 → 最后关闭进程

**服务上线（健康就绪）：**
1. **延迟注册**：服务启动时不立即注册到 Nacos，等待 Spring 容器初始化完成、数据库连接池就绪、健康检查通过后再注册
2. **健康检查通过后才接收流量**：Nacos 配置为只将健康实例纳入负载均衡列表，服务刚启动时处于"初始化中"状态，不会被路由
3. **灰度发布策略**（规划中）：通过 Nacos 的 metadata 实现标签路由，新版本服务标注 `version=v2`，先导入少量流量验证

**Gateway 层防护：**
- Gateway 的路由使用 `lb://service-name` 协议，由 LoadBalancer 从 Nacos 获取健康实例列表
- 当某个实例宕机被剔除后，Gateway 自动将后续请求路由到剩余健康实例
- 配置重试机制：请求失败时自动重试下一个实例

```yaml
spring:
  cloud:
    loadbalancer:
      retry:
        enabled: true
```

**Result（结果）：** 通过主动注销 + 被动检测 + LoadBalancer 缓存刷新的三重机制，确保服务下线后流量在 30 秒内自动切换到健康实例。配合优雅停机，实现服务发布期间对用户无感知的平滑过渡。

## 附录：面试话术技巧

### 如何回答"你遇到过最难的问题是什么？"

> "在做支付模块的时候，遇到过微信支付回调验签始终失败的问题。排查发现是签名原始串的拼接格式与微信官方不一致——缺少了 `&key=` 前缀。这个 Bug 很隐蔽，因为代码看起来'差不多'，但 MD5 对输入极其敏感，差一个字符结果就完全不同。通过逐字比对微信官方文档的签名示例，对比每一步的中间结果，最终定位到差异点。修复后还补上了幂等控制，用 Redis SETNX 防止重复回调。这件事让我深刻认识到：**和外部系统对接时，必须严格按照官方文档的每一个细节实现，不能凭感觉'差不多就行'。**"

### 如何回答"你们项目的技术亮点是什么？"

> "我认为最大的亮点是**在高并发场景下的稳定性设计**。体现在三个层面：第一，网关层统一鉴权 + 限流，在请求入口就过滤掉无效和超限流量；第二，支付模块的双重安全防线——验签防伪造 + 分布式锁防重复，确保资金操作的绝对安全；第三，通过 Seata 和 RocketMQ 分别处理强一致性和最终一致性场景，让不同重要级别的数据采用不同的保障策略，而不是'一刀切'。这些设计都是在真实的生产压力下打磨出来的，不是纸上谈兵。"

### 如何回答"如果让你重新设计这个项目，你会做什么改进？"

> "三点：第一，引入链路追踪（SkyWalking 或 Zipkin），现在出了问题排查效率比较低，需要逐个服务翻日志；第二，完善测试体系，目前缺乏自动化测试，每次改动依赖手动验证；第三，日志集中化，把各服务的日志统一到 ELK，方便全局检索和告警。这些是项目发展到一定规模后必然会做的事情。"
