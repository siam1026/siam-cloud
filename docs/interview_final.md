# Siam-Cloud 架构师面试实战手册（最终纯净版）

> 人设定位：Siam-Cloud 微服务核心架构师 / 高级后端开发
> 项目背景：基于原生 Spring Boot 3.2 + Spring Cloud Alibaba (Nacos, Seata, Sentinel) + Gateway + RocketMQ + ShardingSphere + Redis 打造的高并发外卖/商城平台

---

## 维度一：API 网关与流量治理 (Gateway + Sentinel)

### Q1：你们项目拆分了 10 个微服务，前端请求的鉴权和跨域是怎么统一管理的？

**【情境】：** 在分布式微服务架构下，如果让订单、用户、商品等每个独立服务都去处理 Token 解析和 CORS（跨域），会导致代码严重冗余，且鉴权标准极易出现漏洞。

**【任务】：** 设计一个统一的流量入口，将安全校验、跨域等非业务逻辑彻底下沉到基础设施层。

**【行动】：**
1. 我基于 Spring Cloud Gateway 的 WebFlux 响应式模型构建了统一网关。
2. **跨域控制**：在 Gateway 层面配置 `globalcors`，彻底剥离后端 Provider 的跨域关注点，实现全站统一跨域策略。
3. **全局鉴权**：设计了自定义 `AuthGlobalFilter`。在网关层拦截所有外部请求，校验 Token 的合法性。校验通过后，将 Token 写入 `X-Auth-Token` 请求头中透传给下游服务。对于 `/login`、`/notify`、`/callback` 等接口开放白名单放行。
4. **路由改写**：利用 `StripPrefix=1` 过滤器精准剥离外部访问的前缀（如将 `/api-order/rest/xxx` 剥离为 `/rest/xxx`），实现内外网路径的物理隔离与映射。

**【结果】：** 实现了内部微服务的"无状态化"与"零鉴权负担"，下游服务只需从请求头获取用户信息即可，大幅提升了系统的安全性与横向扩展能力。

---

### Q2：如果遇到大促或者外卖高峰期，网关层如何保护后端微服务不被流量洪峰打垮？

**【情境】：** 午餐高峰期，并发订单请求可能瞬间飙升，如果直接打到订单或商品服务，容易引起数据库连接池耗尽，进而导致全链路雪崩。

**【任务】：** 在网关层建立第一道防线，实现灵活的限流与熔断降级。

**【行动】：**
1. 引入 Sentinel 结合 Gateway 进行入口级别的全局流量控制。
2. 针对核心下单 API 设置基于 QPS 的限流规则（采用令牌桶算法），当请求超过系统承载阈值时，触发快速失败机制。
3. 配置定制化的 `BlockRequestHandler`。在触发限流时，不返回生硬的 500/503 报错，而是返回友好的 JSON 业务提示（如"当前抢单人数过多，请稍后再试"）。
4. Gateway 本身基于 WebFlux 响应式异步模型，相比传统阻塞模型，同等硬件资源下可承载更高的并发连接数。

**【结果】：** 在大流量压测下，系统不仅能够有效削峰填谷，还能保证核心链路（如支付、下单）的资源倾斜，做到了真正的高可用。

## 维度二：高并发支付与资金安全 (Redis 核心场景)

### Q3：外卖系统的支付回调极其关键。如果微信/支付宝服务器因为网络抖动发起了高并发的重复回调，你们如何保证资金安全？

**【情境】：** 第三方支付平台（如微信）在未收到我方明确的 HTTP 200 响应时，会以指数级时间（15s, 30s, 3m...）进行疯狂的重复回调通知。如果不加干预，会导致一笔订单被重复加钱、重复发货。

**【任务】：** 实现绝对的支付回调幂等性控制，且防范外部恶意请求伪造。

**【行动】：** 我利用 Redis 分布式锁设计了防御纵深：

1. **验签前置**：在进入任何业务逻辑前，先调用我底层封装的数字签名校验工具类，严格遵循微信支付 V2 的 MD5 签名规范——将所有参数按字典序排序拼接，末尾拼接 `&key=` + 商户密钥，生成签名后与回调中的 `sign` 字段逐字比对，拦截恶意伪造的虚假支付请求。
2. **SETNX 锁控制**：以 `pay:callback:lock:{订单号}` 作为 Key，执行带有过期时间（设定为 3 分钟，防止服务宕机导致死锁）的 `setIfAbsent` 操作。
3. **拦截与确认**：如果加锁失败，说明该订单的回调正在被处理，系统直接向微信返回 `<return_code>SUCCESS</return_code>`，主动切断微信的后续重试机制。
4. **核心策略（亮点）**：在 `finally` 块中，我故意不主动释放锁，而是让其保留至自然过期。这就完美抵御了微信在极短时间内的并发重发击穿。

**【结果】：** 彻底消除了重复支付回调造成的资金风险，上线至今保证了账务数据的绝对最终一致。

---

### Q4：除了支付回调，Redis 在你们项目中还承载了哪些核心场景？

**【情境】：** 一个成熟的微服务系统，Redis 不能只做简单的 KV 缓存，需要承担更多核心职责。

**【任务】：** 梳理项目中 Redis 的关键应用场景，确保每个场景的选型和实现都经得起生产环境的考验。

**【行动】：**

| 场景 | 数据结构 | 设计要点 |
|------|---------|---------|
| **用户 Session** | String（Key: `LOGIN_USER:{token}`） | 登录成功写入，设置 7 天 TTL；活跃用户持续续期 |
| **支付幂等锁** | String（Key: `pay:callback:lock:{outTradeNo}`） | SETNX 实现，3 分钟过期，不主动删锁 |
| **短信验证码** | String（Key: `sms:code:{mobile}`） | 5 分钟过期，发送时覆盖旧值 |
| **购物车缓存** | Hash（Key: `cart:{userId}`） | 用户加购时先写 Redis，下单时同步到 DB |
| **登录重试限制** | String（Key: `login:lock:{ip}`） | 连续输错密码 5 次后锁定 15 分钟 |

**设计原则：**
1. 所有 Key 统一前缀，便于运维排查和管理
2. 过期时间必须有，防止内存泄漏
3. 大 Key 预警：购物车超过 100 个商品时异步刷盘
4. 容灾降级：Redis 不可用时，Session 查询降级为查 DB

**【结果】：** Redis 在项目中承担了认证、幂等、缓存、限流等多重职责，每个场景根据业务特性选择了合适的数据结构，配合 TTL、前缀规范、容灾降级策略，确保在生产环境下稳定运行。

## 维度三：分布式事务与异步流转 (Seata + RocketMQ)

### Q5：用户下单时，需要同时调用订单服务创单、商品服务扣库存、促销服务核销优惠券。如果中途报错，怎么保证数据的一致性？

**【情境】：** 在微服务架构下，本地的 `@Transactional` 只能管辖单一数据库。跨微服务的 RPC 调用一旦出现网络超时或下游报错，极易产生部分成功、部分失败的"脏数据"（如库存扣了但订单没生成）。

**【任务】：** 在保证系统吞吐量的前提下，实现跨微服务下单链路的强一致性。

**【行动】：** 我主导引入了 Seata 的 AT（Automatic Transaction）模式：

1. 以外部网关的下单请求为起点，在 Order 服务的发起端打上 `@GlobalTransactional` 注解，向 TC（事务协调器）注册全局事务 XID。
2. Seata 会在底层自动拦截各个服务的业务 SQL，在执行业务操作前，生成包含修改前和修改后数据的 `undo_log`（回滚日志），并与业务数据在同一个本地事务中提交。
3. 如果调用链上的任何一个节点抛出异常，TC 会立即向所有参与者下发回滚指令，各节点利用 `undo_log` 中的前置镜像生成反向 SQL 进行自动补偿。
4. **一致性级别权衡**：资金相关和核心业务（余额扣减、库存扣减、订单创建）走强一致性（Seata）；非核心链路（通知、日志、统计）走最终一致性（RocketMQ）。

**【结果】：** 以极小的代码侵入性（仅加注解），完美保障了核心下单链路的强一致性，有效防止了外卖平台的超卖事故。

### Q6：下单支付成功后，系统还要发短信、给骑手派单、更新报表。如果这些都写在主流程里，响应太慢怎么办？

**【情境】：** 支付成功后的后置动作繁多，导致核心接口响应时间长（RT），且一旦边缘接口（如发短信）挂了，会导致整个主流程报错回滚，系统耦合度极高。

**【任务】：** 对核心链路进行深度解耦，提升吞吐量与容错率。

**【行动】：** 采用 RocketMQ 事务消息实现最终一致性：

1. 支付服务在确认收款后，首先通过 RocketMQ 发送一条半消息（Half Message）到 Broker，此时消息对消费者不可见。
2. 执行本地事务（修改订单状态为"已支付"，记录支付流水）。
3. 本地事务成功后，向 Broker 发送 Commit 指令，此时消息才真正投递给下游（短信、派单系统）进行异步消费。
4. **容错机制**：万一支付服务在发送 Commit 前宕机，MQ 的回查机制会反向调用我实现的 `checkLocalTransaction` 接口，检查数据库订单状态，决定是 Commit 还是 Rollback。

**【结果】：** 核心支付接口的 RT 从 1.5 秒压降到了 200 毫秒以内，极大地提升了用户体验，并实现了各业务域的物理隔离。

## 维度四：海量数据路由与分片 (ShardingSphere)

### Q7：随着系统订单量暴增，单表千万级数据导致查询极慢，你是如何设计数据库存储架构的？

**【情境】：** 外卖系统的历史订单数据增长极快，MySQL 单库单表（tb_order）很快遇到了 B+ 树索引层级过深、I/O 遇到瓶颈的严重性能衰退问题。

**【任务】：** 在不改变上层 Java 业务代码（依然使用 MyBatis-Plus 标准写法）的前提下，实现数据库的水平无缝扩容。

**【行动】：** 我引入了 Apache ShardingSphere-JDBC 作为数据层的智能路由引擎：

1. **分片键选择**：经过对慢 SQL 频次的分析，发现绝大部分高频查询是"C 端用户查自己的订单"。因此我选取 `member_id` 作为分库键，`order_id` 作为分表键。
2. **路由算法设计**：采用标准 Inline 表达式。将数据库水平拆分为 `siam_cloud_0` 和 `siam_cloud_1`（路由逻辑：`member_id % 2`），单库内再分为 `tb_order_0` 和 `tb_order_1`（路由逻辑：`order_id % 2`）。
3. **分布式主键**：弃用数据库自增 ID，全面接入雪花算法（Snowflake）生成趋势递增且全局唯一的订单 ID，防止数据合并时的主键冲突。

**【结果】：** 将单表 I/O 压力完美均摊到多个物理节点，系统轻松支撑亿级订单数据。上层微服务对底层的分表完全无感知，实现了架构的平滑扩容。

---

### Q8：分库分表后，分页查询和跨分片统计怎么解决？

**【情境】：** 分库分表后，跨分片的分页查询性能很差，因为需要聚合所有分片的结果再排序。管理后台的订单列表和统计数据面临挑战。

**【任务】：** 在分片架构下保障管理端查询效率和数据准确性。

**【行动】：**
1. **分页查询优化**：管理后台的分页查询一般不跨分片——按商家维度查询（商家 id 固定），天然路由到单一分片。对于需要全局数据的场景，限制最大页数（如最多翻 100 页）。
2. **统计数据预聚合**：对于大数据量统计（如今日交易额、订单数），通过定时任务预聚合到统计表中，查询时直接读统计表而非实时扫分片。
3. **主键全局唯一**：使用雪花算法确保各分片的主键不冲突，便于后续可能的数据迁移和合并。

**【结果】：** 管理端查询响应时间控制在 200ms 以内，统计数据通过预聚合避免了实时跨分片扫描带来的性能开销。

## 维度五：服务治理与 RPC 通信 (Nacos + Feign)

### Q9：为什么选用 Nacos 作为基础设施？在服务上下线时，如何避免流量打到已经挂掉的节点？

**【情境】：** 微服务节点众多，需要健壮的注册中心和配置中心。同时，服务重启、发布、扩容时，如果负载均衡器仍将请求路由到已下线的实例，会导致大量请求失败。

**【任务】：** 构建稳固的服务发现机制，实现优雅上下线和零宕机发布。

**【行动】：**

1. **基础设施选型**：选用 Nacos 2.x，因为它天然集成了 Discovery（注册）和 Config（配置），支持长轮询机制，比传统的 Eureka 具有更快的上下线感知速度。Nacos 同时支持 AP 和 CP 模式，我们默认使用 AP 模式保障服务发现的高可用。

2. **服务下线（优雅停机）**：
   - 主动注销：服务关闭前主动向 Nacos 发起注销请求，标记自己为不健康状态
   - 被动检测：Nacos 心跳检测发现服务超过 15 秒无心跳，自动将其标记为不健康并剔除
   - LoadBalancer 缓存刷新：客户端缓存的服务实例列表有 TTL（默认 30 秒），超时后重新从 Nacos 拉取，自动剔除不健康实例

3. **服务上线（健康就绪）**：
   - 延迟注册：服务启动时不立即注册，等待 Spring 容器初始化完成、数据库连接池就绪后再注册
   - 健康检查通过后才接收流量：Nacos 只将健康实例纳入负载均衡列表

4. **Gateway 层防护**：路由使用 `lb://service-name` 协议，当某个实例宕机被剔除后，Gateway 自动将后续请求路由到剩余健康实例。

**【结果】：** 通过主动注销 + 被动检测 + LoadBalancer 缓存刷新的三重机制，确保服务下线后流量在 30 秒内自动切换到健康实例，实现服务发布期间对用户无感知的平滑过渡。

### Q10：在 RPC 服务间调用（Feign）中，你做过哪些核心的容错设计？

**【情境】：** 微服务间的 Feign 调用经常因为网络抖动、下游服务未启动、或者序列化问题引发各种异常。尤其是下游返回的包装类 `BasicResult<T>` 中，`getData()` 可能为 null，如果业务层直接拆箱为 `int` 或 `BigDecimal`，会导致连环 `NullPointerException`。

**【任务】：** 彻底扫除 Feign 调用的隐藏地雷，增强 RPC 调用的鲁棒性。

**【行动】：**

1. **Feign 拆箱安全防范（亮点）**：我们的 RPC 接口统一返回了带有泛型的 `BasicResult<T>` 包装类。我发现在复杂网络下，如果下游服务未启动或发生熔断，返回的 `.getData()` 极易为 null。如果业务层直接拆箱，会导致连环 NullPointerException。为此，我在底层架构中统一封装了安全拆箱工具方法（`safeInt`、`safeBigDecimal`），拦截所有非法 null 值并赋予合理的业务默认值（如 0 或 `BigDecimal.ZERO`），全局覆盖了 60+ 处调用点。

2. **Token 隐式透传**：通过实现 `RequestInterceptor`，在 Feign 发起调用前，自动将上下文中的 `X-Auth-Token` 提取并注入到新的 Request Header 中，保证了用户登录态在微服务链路中的无缝穿透。

3. **超时配置**：为 Feign 配置了合理的连接超时（5 秒）和读取超时（10 秒），防止下游服务响应慢导致调用方线程池耗尽。

**【结果】：** 消除了至少 60 多处潜在的空指针隐患，极大增强了 RPC 调用的鲁棒性，系统的联调效率提升了 40%。

## 附录：高频面试话术

### 如何回答"你遇到过最难的问题是什么？"

> "在做支付模块的时候，微信支付回调的验签和幂等控制是两个关键挑战。验签方面，微信支付 V2 使用 MD5 签名，要求将所有参数按字典序排序后拼接 `&key=` + 商户密钥来计算签名。这个环节对输入极其敏感，差一个字符结果就完全不同。我通过逐字比对微信官方文档的签名示例，确保了验签逻辑的精确性。幂等方面，微信在网络抖动时会重复发送同一笔订单的回调，我用 Redis SETNX 实现了分布式锁，以订单号为 Key，3 分钟过期，并且在 finally 中故意不主动释放锁，让其保留至自然过期，完美抵御了微信在极短时间内的并发重发击穿。这件事让我深刻认识到：**和外部系统对接时，必须严格按照官方文档的每一个细节实现，不能凭感觉'差不多就行'。**"

### 如何回答"你们项目的技术亮点是什么？"

> "我认为最大的亮点是**在高并发场景下的稳定性设计**。体现在三个层面：第一，网关层统一鉴权 + 限流，在请求入口就过滤掉无效和超限流量；第二，支付模块的双重安全防线——验签防伪造 + 分布式锁防重复，确保资金操作的绝对安全；第三，通过 Seata 和 RocketMQ 分别处理强一致性和最终一致性场景，让不同重要级别的数据采用不同的保障策略，而不是'一刀切'。这些设计都是在真实的生产压力下打磨出来的，不是纸上谈兵。"

### 如何回答"你觉得这个项目的不足在哪里？如果让你重新设计会做什么改进？"

> "三点：第一，引入链路追踪（SkyWalking 或 Zipkin），现在出了问题排查效率比较低，需要逐个服务翻日志；第二，完善测试体系，目前缺乏自动化测试，每次改动依赖手动验证；第三，日志集中化，把各服务的日志统一到 ELK，方便全局检索和告警。这些是项目发展到一定规模后必然会做的事情。"

### 如何回答"你做这个项目最大的收获是什么？"

> "最大的收获是从 0 到 1 落地了一套高可用的微服务架构，并在实践中深刻理解了分布式事务（Seata）、分库分表（ShardingSphere）和网关流量治理在复杂系统中的重要性。之前这些技术对我来说更多是理论概念，通过这个项目，我真正理解了 Seata AT 模式的 undo_log 机制、ShardingSphere 的分片路由原理、以及 Gateway 响应式模型与传统 Servlet 的本质区别。这种从理论到实践的跨越，是我最大的成长。"
